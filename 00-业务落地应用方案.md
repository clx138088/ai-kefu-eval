# 评估体系在客服业务系统中的落地应用

> 回答一个核心问题：这套评估框架不只是"面试作品"，它在真实的客服业务中怎么跑起来？

---

## 一、总体架构

```
┌─────────────────────────────────────────────────────────────────┐
│                     AI 客服质量保障体系                             │
│                                                                 │
│  ┌──────────────┐   ┌──────────────┐   ┌──────────────────────┐ │
│  │  Layer 1     │   │  Layer 2     │   │  Layer 3             │ │
│  │  离线评估     │   │  在线拦截     │   │  持续监控             │ │
│  │             │   │             │   │                      │ │
│  │  上线前     │   │  对话进行中   │   │  上线后               │ │
│  │  版本迭代前  │   │  实时决策     │   │  日常运营             │ │
│  └──────┬───────┘   └──────┬───────┘   └──────────┬───────────┘ │
│         │                  │                      │             │
│         ▼                  ▼                      ▼             │
│   评测集 + Judge      规则引擎 + 三问法        北极星指标 + 抽检    │
```

三层不是串行关系，而是一个闭环——Layer 3 的异常信号触发 Layer 1 的定向评估，Layer 1 的发现更新 Layer 2 的拦截规则。

## 二、Layer 1：离线评估（上线前的质量门控）

### 2.1 什么时候触发

| 触发场景 | 频率 | 执行人 |
|----------|------|--------|
| Bot 新版本上线前 | 每次发版 | QA / AI PM |
| Bot prompt 或知识库有重大更新 | 每次更新 | AI PM |
| 新增意图类型或场景 | 按需 | AI PM |
| Layer 3 异常信号触发抽检 | 按需 | 运营 |
| 定期全量巡检 | 每月一次 | QA |

### 2.2 执行流程

```
Step 1: 更新评测集
  ├─ 从真实客服日志中采样新的用户 query
  ├─ 针对上一周期暴露的问题类型定向构造 case
  └─ 人工标注预期意图

Step 2: 跑 Judge
  ├─ 将评测集导入 Coze Workflow（或自建 API）
  ├─ 跑 Judge Prompt（选择严格度版本：日常用 v3.1，发版前用 v3.1s）
  └─ 生成 Tracing 表

Step 3: 对比分析
  ├─ 计算 E 率 / D2 率 / 分类分布
  ├─ 与上一版本做改前改后对比
  └─ 生成评估报告

Step 4: 门控决策
  ├─ D2 > 0 → ❌ 拦截上线，修复后重跑
  ├─ C2 新增 > 3 条 → ⚠️ 建议修复后上线
  └─ D2 = 0 且 E 率不下降 → ✅ 通过门控
```

### 2.3 门控标准（建议）

| 指标 | 拦截阈值 | 警告阈值 | 说明 |
|------|----------|----------|------|
| D2 | > 0 | — | 一票否决。任何越权回答都不允许上线 |
| C2 新增 | > 5 | > 2 | 新增的入口/链接缺失 |
| E 率下降 | > 5pp | > 2pp | 回答质量退化 |
| A 类新增 | > 3 | > 1 | 意图理解回退 |

### 2.4 业务系统中的嵌入位置

```
CI/CD Pipeline:
  代码提交 → 构建 → 部署到 staging → [离线评估 ← 在这里] → 门控通过 → 发布到 production
```

不是手动跑——接入 CI/CD，staging 部署后自动触发评估脚本，门控不通过则阻断发布。

## 三、Layer 2：在线拦截（对话进行中的实时决策）

### 3.1 在 Bot 架构中的位置

```
用户消息
  ↓
前置过滤（敏感词/垃圾消息）
  ↓
意图分类（现有）
  ↓
┌─────────────────────────────┐
│  🆕 风险分级引擎（P0）        │
│                             │
│  输入：user_query + intent  │
│  输出：HIGH / LOW           │
│                             │
│  五条红线串行检查：          │
│   ① 人身安全/健康           │
│   ② 赔付/退款/补发          │
│   ③ 情绪信号强烈            │
│   ④ 需联系外部方            │
│   ⑤ 法律边界               │
└──────────────┬──────────────┘
               │
     ┌─────────┴─────────┐
     │                   │
   HIGH                 LOW
     │                   │
     ▼                   ▼
  过渡语 +           Bot 生成回答
  转人工队列              │
                     ┌───┴───┐
                     │ 三问法  │
                     │ 自检   │
                     └───┬───┘
                    通过  │  拦截
                     ▼    │
                   输出   ─→ 过渡语 + 转人工
```

### 3.2 实现方式

**方式一：前置规则引擎（推荐）**

在 Bot 服务的中间件层实现，Bot 框架（如 Coze/Dify/LangChain）的 hook 机制：

```python
# 伪代码：在 Bot 的 pre_response hook 中
def pre_response_hook(query: str, intent: str, context: dict):
    risk = risk_classifier.classify(query, intent)
    if risk.level == "HIGH":
        return {
            "block": True,
            "transition_msg": risk.transition_msg,
            "route_to": "human_queue",
            "tags": risk.tags  # 如 ["赔付", "人身安全"]
        }
    return {"block": False}
```

部署后 Bot 服务对每个 query 增加 ~5ms 延迟（关键词匹配，无 LLM 调用）。

**方式二：Bot Prompt 内置（备选）**

如果不能改 Bot 中间件，在 System Prompt 中嵌入：

```
在生成回答之前，先检查用户 query 是否触发以下任一条件。
如果触发 → 不要生成回答，改为输出 [TRANSFER: 原因]：
  - 涉及人身安全/健康（宝宝/孕妇/儿童/过敏/起红点/安全/皮肤）
  - 涉及赔付/退款/补发（退款/补发/赔偿/漏了/少发/破损）
  - 用户情绪强烈（投诉/曝光/差评）
  - 需要联系外部方才能处理（物流核实/仓库查询）
  - 涉及法律合规边界（多开发票/虚开）
```

方式一更可靠（不依赖 LLM 的指令遵循能力），方式二实现更快（改一行 prompt）。

### 3.3 转人工后的处理

```
用户看到过渡语 → 进入人工队列 → 人工客服看到：
  ┌──────────────────────────┐
  │  用户 query：洗衣液漏了    │
  │  意图：S-05 破损          │
  │  风险标签：🔴 赔付退款     │
  │  拦截规则：②              │
  │  建议处理：引导拍照→核实→补发/退款 │
  └──────────────────────────┘
```

人工客服不是从零开始——系统已经告诉 ta 用户遇到了什么问题、可能是什么原因、建议怎么处理。这减少了人工客服的"理解成本"，提升了处理效率。

## 四、Layer 3：持续监控（上线后的异常发现）

### 4.1 北极星指标

| 指标 | 定义 | 异常阈值 | 说明 |
|------|------|----------|------|
| 转人工率 | 被规则引擎拦截的 query 占比 | 突降 >30% | 突降可能意味着规则被绕过 |
| 二次进线率 | 同一用户 24h 内再次咨询的比例 | 突升 >20% | 首次回答没解决问题 |
| 客诉率 | 含"投诉/差评/曝光"的 query 占比 | 突升 >50% | 整体体验恶化 |
| AI 回答率 | Bot 实际生成回答的 query 占比 | 突升 >20% | 可能规则引擎失效 |

### 4.2 触发式抽检

```
北极星指标异常
  ↓
自动拉取异常时间窗口内的对话样本（50-100条）
  ↓
跑离线评估（Layer 1）
  ↓
定位问题 → 更新评测集 → 修复 → 验证
```

### 4.3 与客服运营 Dashboard 的集成

现有的客服 Dashboard 通常是这样的：
- 会话量、响应时长、满意度

加入评估体系后新增：
- **AI 质量分**（E 率，每周更新）
- **风险拦截量**（D2 拦截数，实时）
- **拦截原因分布**（①②③④⑤ 各触发多少次）
- **待修复问题清单**（按优先级排序）

## 六、不同规模团队的落地路径

### 6.1 小团队（1-3 人，没专门 QA）

```
第一周：Layer 2（在线拦截）
  → 改一行 Bot prompt，加入五条红线自检
  → 成本最低，见效最快（D2 立刻归零）

第二周：Layer 1（离线评估）
  → 用现有 91 条评测集 + Judge Prompt
  → 每次发版前手动跑一次，Excel 记录对比

一个月后：Layer 3（持续监控）
  → 拉客服系统的转人工率 + 二次进线率
  → 每周看一眼，异常了就抽检
```

### 6.2 中型团队（5-15 人，有 QA + 运营）

```
Layer 2：规则引擎中间件
  → 不是 prompt 而是代码实现，有灰度发布、有回滚开关

Layer 1：自动化评估 Pipeline
  → CI/CD 集成，staging 部署后自动触发
  → 门控不通过自动阻断发布

Layer 3：监控 Dashboard
  → BI 工具（如 Metabase/Grafana）拉实时数据
  → 异常告警 → 自动拉样本 → 人工抽检确认
```

### 6.3 大型团队（15+ 人，有专门 AI 质量团队）

```
全部三层 + 以下增强：

评测集管理平台：
  → 不是 markdown 文件，是在线标注+版本管理+协作审核

Judge 多模型投票：
  → 每条 case 用 2-3 个不同 LLM 独立判
  → 不一致的 case 自动标记为"需人工仲裁"

A/B 实验：
  → 新 Bot 版本 10% 流量 vs 旧版本 90% 流量
  → Layer 3 实时对比两组的转人工率、满意度

对抗样本自动生成：
  → 用 LLM 自动改写评测集中的 query（同义替换、加噪声、多语言）
  → 评测集从 91 条自动扩展到 500+ 条
```

## 七、落地的关键风险与应对

| 风险 | 影响 | 应对 |
|------|------|------|
| 规则引擎误拦截 FAQ 型 query | L 类"我的快递到哪了"被④拦截 | ④增加 intent 白名单——L 类目中"进度查询"子意图不拦截 |
| Bot 回答变好但 E 率反而下降 | STAR-S 严格阈值捕获更多 C2 | 区分"内容 C2"和"入口 C2"，后者可接受 |
| 转人工量激增 | 人工客服接不过来 | 先上 P0 规则引擎 → 观察实际转人工量 → 再决定是否需要加人 |
| 人工标注和 LLM Judge 持续不一致 | 评估体系公信力下降 | 每月做一次 Kappa 校验，边界 case 加入锚点库 |

## 八、与面试的关系

如果面试官问"你这个项目怎么落地"，不要只回答"三层架构"——要讲清楚**推进节奏**：

> "落地的关键是分层推进，不是一口气全上。先花 10 分钟改一行 Bot prompt 把 D2 拦截掉——这是最快见效的。然后搭离线评估 Pipeline，确保每次发版前有质量门控。最后才是持续监控 Dashboard。三层之间是有优先级和依赖关系的——Layer 2 保安全底线，Layer 1 保迭代质量，Layer 3 保长期运营。"

**三个"10 分钟"演示**（如果面试官让你现场演示思路）：

1. Layer 2 最快路径：改 Bot System Prompt → 加入五条红线 → 部署 → D2 归零
2. Layer 1 最快路径：拿现有 91 条评测集 → 跑一次 Judge → 生成评估报告
3. Layer 3 最快路径：拉客服系统后台 → 导出近 7 天对话 → 抽样 50 条 → 跑 Judge → 出周报
