# 人工标注 vs LLM Judge 差异分析

> 对比对象：  
> 
> - **人工**：使用 MECE v3.0 框架对 50 条旧基线的手工分类  
> - **LLM**：Coze Workflow 批量跑 Judge Prompt v3.0 对同一批 50 条 case 的分类  

---

## 一、最重要的前置发现：两者评估的是不同的 AI 回答

**人工标注**基于的是"模拟基线 Bot"的回答——人工按照假设的关键词匹配+人工根据FAQ模板模拟出的回答。详见 `04-tracing表+基线评分模板.md` 中的假设：

> "意图分类目前使用关键词匹配 + 简单规则，不支持多意图。检索方式为关键词匹配 FAQ 标题，无 query rewrite。"

**LLM Judge** 评估的是 **Coze 平台上实际搭建的 Bot** 的真实输出。Coze Bot 的回答质量远高于人工模拟的基线 Bot。

这意味着很多分类差异不是因为 Judge 标准不同，而是因为**被评估的回答本身就不同**。Coze Bot 确实比人工假设的"关键词匹配系统"回答得好得多。

---

## 二、整体分布对比（50 条重叠 case）

| 分类  | 人工 MECE v3.0 | LLM Judge | 差异      |
| --- | ------------ | --------- | ------- |
| E   | 18 (36%)     | 41 (82%)  | **+23** |
| A1  | 2            | 0         | -2      |
| A2  | 5            | 1         | -4      |
| A3  | 2            | 0         | -2      |
| B2  | 1            | 0         | -1      |
| B3  | 0            | 1         | +1      |
| C2  | 13           | 0         | **-13** |
| C3  | 4            | 0         | -4      |
| D1  | 2            | 0         | -2      |
| D2  | 3            | 7         | **+4**  |

### 核心差异总结

| 偏差方向        | 幅度   | 原因                                           |
| ----------- | ---- | -------------------------------------------- |
| **E 大幅增加**  | +23条 | Coze Bot 实际回答质量远高于人工模拟的基线 Bot                |
| **C 类归零**   | -17条 | ①Coze Bot 回答更完整 ②LLM Judge 对 C 类（信息缺失）判据更宽松  |
| **A 类大幅减少** | -7条  | ①Coze Bot 多意图处理能力更好 ②人工标注时的"模拟答案"本身就故意漏答了    |
| **D2 增加**   | +4条  | LLM Judge 对 Step 4 判据执行更严格——S类 case 更容易触发 D2 |

## 三、关键分歧 Case 逐条分析

### 🔴 严重分歧（分类完全不同）

#### 1. P-10：这个参加满减吗，什么时候能发货（多意图）

| 维度         | 内容                                                                              |
| ---------- | ------------------------------------------------------------------------------- |
| 人工分类       | **A2** — "多意图只答一半"（只答了满减，没提发货）                                                  |
| LLM 分类     | **E** — 全部回应+捕捉隐含诉求                                                             |
| 实际 Coze 回答 | "咱们商城当前在售的商品都参与官方满减活动哦，您下单付款后我们会在48小时内为您安排发货，要是您的收货地址处于疫情管控区域的话，发货时间可能会稍有延迟…"   |
| 分析         | **LLM 正确**。Coze Bot 的回答确实同时覆盖了"满减"和"发货"两个显性诉求。人工标注时使用的是模拟回答（"只答了满减规则"），不是同一个回答。 |

#### 2. P-09：洗衣液能洗羊毛衫吗（涉及材质风险）

| 维度         | 内容                                                                                              |
| ---------- | ----------------------------------------------------------------------------------------------- |
| 人工分类       | **A3** — 问"能不能"答"怎么洗"，意图域不同（对话记录记载：从 B3 修正为 A3）                                                 |
| LLM 分类     | **E** — 全部回应+捕捉隐含诉求                                                                             |
| 实际 Coze 回答 | "咱们家中性配方的洗衣液（比如手洗专用款、亮白增艳款）是可以洗羊毛衫的，温和不损伤羊毛纤维，普通碱性款洗衣液就不建议用来洗哦。另外您可以先看下羊毛衫的洗水标…水洗的话用常温水轻柔按压就好…" |
| 分析         | **LLM 正确**。Coze Bot 的回答直接回应了"能不能"（可以，但分中性/碱性）+ 后续还给了具体操作建议。人工标注时的模拟回答只说"可以手洗"未做区分，是不同质量的回答。     |

#### 3. S-01：洗衣液漏了，快递箱都是湿的

| 维度               | 内容                                                                                                             |
| ---------------- | -------------------------------------------------------------------------------------------------------------- |
| 人工旧版 v2.0        | 19/19 满分，兜底=1（"正确转人工处理"）→ 映射为 **E**                                                                            |
| 人工 MECE v3.0 重分类 | 对话记录记载为 **D2**（"修正了 S-01（E→D2）"——即从旧版的 E 改为 D2）                                                                |
| LLM 分类           | **D2** — 涉及赔付/退款/补发+需联系外部方                                                                                     |
| 分析               | **一致（D2）**。人工在 MECE v3.0 框架下也修正为 D2。说明 MECE v3.0 的 D 类判据比旧版更严格——旧版认为"引导拍照+转人工=正确"，新版认为"AI 在转人工之前承诺了赔付方案 = 越权"。 |

#### 4. L-07：快递显示签收但我没收到

| 维度         | 内容                                                                                                                                                 |
| ---------- | -------------------------------------------------------------------------------------------------------------------------------------------------- |
| 人工旧版 v2.0  | 兜底=0，badcase="该转未转"（高风险涉及丢件/冒领）→ 映射为 **D2**                                                                                                        |
| LLM 分类     | **E** — "回答为合规的问题处理流程引导，未越权处理敏感事项"                                                                                                                 |
| 实际 Coze 回答 | "您可以先检查下快递柜、物业前台、小区代收点…要是以上地方都没找到的话，请您联系我们提供对应物流单号，我们会马上对接快递员核实签收情况，确认非您本人或授权人员签收的话，我们会立即启动丢件流程…"                                                  |
| 分析         | **争议较大**。LLM 认为这是"流程引导"而非越权承诺，判 E。但按 MECE v3.0 Step 4 的判据④（AI 无法自行完成，需联系外部方——对接快递员），这个回答确实涉及了需联系外部方的操作。**建议人工复核确认**：是 E（正确流程引导）还是 D2（涉及外部方不应自动回答）。 |

#### 5. A-03：为什么我的优惠券用不了

| 维度         | 内容                                                                                             |
| ---------- | ---------------------------------------------------------------------------------------------- |
| 人工旧版 v2.0  | 兜底=0，badcase="该转未转"（涉及用户实际权益，无法精准排查时应转人工）                                                      |
| LLM 分类     | **E** — 全部回应+捕捉隐含诉求                                                                            |

## 四、系统性偏差分析

### 偏差 1：LLM Judge 对 C 类（信息缺失）几乎"失明"

| 指标    | 人工          | LLM         |
| ----- | ----------- | ----------- |
| C 类总计 | 17/50 (34%) | 1/91 (1.1%) |

这不是因为 Coze Bot 回答更完整就能完全解释的。即使考虑 Coze Bot 质量提升，C 类从 34% 降到接近 0% 也说明 **LLM Judge 的 Step 3 判据执行过于宽松**。

**可能原因**：LLM 更容易"脑补"缺失的信息——它看到回答给出了大致方向，就认为"用户能完成操作"，但实际用户可能还需要具体的链接、入口、时效等信息。

**建议**：在 Judge Prompt 的 Step 3 中增加更具体的检查项，比如：

- "回答中是否包含了可点击的链接/按钮/入口？"
- "回答中是否给出了明确的时效/时间承诺？"
- "回答中是否给出了操作失败后的备选方案？"

---

### 偏差 2：LLM Judge 对 D2 的判据执行更"机械化"

LLM 找到了 19 条 D2（包括 CSV 新增 case），人工在 50 条中只找到 3+2=5 条 D 类。即使考虑 Coze Bot 回答更完整导致更多 case 通过了 Step 1-3 从而暴露了 Step 4 的问题，这个差异仍然显著。

**关键分歧点在于**：AI 回答中提到"帮您联系/对接/核实"到底算不算"需联系外部方"触发 D2？

- LLM 倾向于：只要提到"联系快递/仓库/财务"就触发 D2
- 人工倾向于：如果是"引导用户提供信息→人工客服来操作"的流程说明，不算越权

这需要在 Judge Prompt 中明确边界。

---

### 偏差 3：A 类差异主要由回答质量差异导致

人工标注的 A 类（9条）中，大部分是因为"模拟 Bot 不支持多意图"。Coze Bot 实际能处理多意图，所以 LLM 判这些为 E。这是合理的——Judge 评的是回答本身，不是系统的能力上限。

## 五、调整方向

| 优先级    | 行动                           | 说明                                                        |
| ------ | ---------------------------- | --------------------------------------------------------- |
| **P0** | 人工复核 L-07、S-09 的 D2 判定       | 这两个 case 在 D2 vs E 的边界上，LLM 和人工有明显分歧                      |
| **P1** | 加强 Judge Prompt Step 3 的检查粒度 | LLM 对 C 类几近失明，需要增加更具体的完整性检查项                              |
| **P1** | 明确 D2 判据④的边界                 | "AI 提到需联系外部方"和"AI 正确引导用户走人工处理流程"的区分需写清楚                   |
| **P2** | 对 P-04 的双触发 case 做规范         | B 类和 D 类同时触发时的优先级规则（当前决策树 Step 2 先于 Step 4，所以 B 类优先输出是对的） |

---

## 六、结论

1. **大部分差异是由不同的 AI 回答质量导致的，不是 Judge 标准本身的问题。** 人工标注用的是模拟回答，LLM 评的是 Coze Bot 真实输出。Coze Bot 确实回答得更好。

2. **LLM Judge 整体判分偏宽松**（E 率 82% vs 人工 36%），尤其对信息完整性（C 类）敏感度不够。

3. **在 D2 判据上，LLM 比人工更严格/更机械化**，这导致 S 类 case 的 D2 率高达 60%。

4. **少数 case（L-07、S-09）存在实质性判断分歧**，建议人工复核后作为锚点 case 写入 Judge Prompt。
