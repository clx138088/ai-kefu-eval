# AI 智能客服质量评估体系 — 企业落地方案

---

## 一、项目背景与目标

### 1.1 业务问题

企业 AI 客服 Bot 上线前，产品团队面临一个核心决策问题：**该 Bot 能否上线？**

传统质量评估依赖人工抽样打分，存在三个结构性缺陷：

| 缺陷     | 表现                                 | 后果       |
| ------ | ---------------------------------- | -------- |
| 总分掩盖根因 | 一条回答获评"3 分"，但无法定位具体错在哪个环节          | 改进方向不明确  |
| 维度不独立  | 意图理解错误导致下游信息缺失，同一错误在多个维度重复扣分       | 评估失真     |
| 量纲不可比  | "意图理解"与"操作指引"属于不同评价维度，简单加权相加缺乏统计意义 | 分数不可横向对比 |

### 1.2 项目目标

构建一套**可复现、可审计、可驱动改进**的 AI Agent 回答质量评估体系，实现：

- **评估自动化**：将人工评估从 4.5 小时/轮压缩至 3 分钟/轮
- **根因可追溯**：每条评估结果携带完整决策路径，支持团队 Review
- **改进可量化**：评估分类直接映射改进方向，形成"评估 → 诊断 → 改进 → 验证"闭环

### 1.3 适用范围

本方案虽以电商客服为验证场景，但评估方法论适用于任意企业级 AI Agent 质量审计场景——包括但不限于智能客服、知识库问答、Agent 任务执行等。

## 三、Layer 1：离线评估系统

### 3.1 系统定位

离线评估是 AI 客服上线前的**质量门控**——每次 Bot 版本更新或知识库变更后，自动触发全量评估，不通过门控标准则阻断上线。

### 3.2 技术实现

**平台选型**：Coze（字节跳动 Agent 平台）
**核心架构**：Workflow 双节点编排 — Bot 节点（RAG + 知识库检索）→ Judge 节点（决策树评估）
**评测集规模**：91 条，覆盖 5 大类目（商品咨询、活动规则、物流查询、发票开具、售后投诉），**91 条均成功获得 Bot 有效回答**（无空响应或生成失败）

**执行流程**：

```
Step 1 — 评测集准备
  ├─ 从实际客服场景构建意图 Taxonomy（5 大类目 × 多级子场景）
  ├─ 搭建 91 条评测集（含 user_query + 预期意图标注）
  └─ 30 条人工标注用于 Judge 校准基线

Step 2 — 自动化跑评
  ├─ Coze Workflow 逐条执行：Bot 检索知识库 → 生成回答 → Judge 决策树诊断
  ├─ Bot 挂载 6 个知识库文档（商品信息/活动规则/物流FAQ/发票FAQ/售后FAQ/Bot人设）
  └─ 每条输出：classification + decision_path + reason

Step 3 — 对比分析
  ├─ 计算 E 率 / D2 率 / 分类分布
  ├─ 人工标注 30 条 vs Judge 判定交叉对比 → 定位系统性偏差
  └─ 生成 Tracing 表 + 评估报告

Step 4 — 门控决策
  ├─ D2 > 0 → 拦截上线（一票否决）
  ├─ B 类 > 0 → 拦截上线（存在事实错误或幻觉）
  ├─ E 率下降 > 5pp → 警告（需排查退化原因）
  └─ D2 = 0 且 E 率不下降 → 通过门控
```

### 3.3 门控标准

| 指标            | 拦截阈值  | 警告阈值  | 判定逻辑                          |
| ------------- |:-----:|:-----:| ----------------------------- |
| D2（越权回答）      | > 0   | —     | 一票否决。AI 在不应回答的场景越权作答，涉及安全合规风险 |
| B 类（事实错误/幻觉）  | > 0   | —     | 存在不可接受的错误信息输出                 |
| C2 新增（操作入口缺失） | > 5   | > 2   | 影响用户实际操作体验                    |
| E 率下降         | > 5pp | > 2pp | 整体回答质量显著退化                    |
| A 类新增（意图理解错误） | > 3   | > 1   | 核心理解能力回退                      |

### 3.4 Judge 校准机制

LLM Judge 并非开箱即用。项目通过三轮校准，将 Judge 判定一致性提升至可用水平：

| 校准轮次         | 发现的系统性偏差                | 根因分析                        | 修复方法                                |
| ------------ | ----------------------- | --------------------------- | ----------------------------------- |
| v3.0 → v3.1  | C 类检测过弱（1.1% vs 人工 34%） | LLM 对信息缺失天然"失明"，倾向认为给定信息已足够 | Step 3 从开放式判断改为 STAR 结构化清单          |
| v3.0 → v3.1  | D2 判据边界模糊               | 未区分"仅查询物流状态"与"需要人工介入"       | Step 4 引入 Q0 前置原则（先判断 AI 是否有能力回答）   |
| v3.0 → v3.1  | B/D 双触发无优先级             | 同一 case 同时触发两类，主分类不确定       | 确立 B 优先原则（事实错误是不管谁回答都错的）            |
| v3.1 → v3.1s | A1 判据过宽                 | LLM 将"无个性化"等同于"意图理解不足"      | 收紧为需同时满足"query 有个性化信号 + Bot 只给通用规则" |

**核心认知**：将 LLM Judge 视为需要定期校准的测量仪器，而非可直接信任的裁判。校准方法为"人工标注 → 交叉对比 → 定位偏差 → 修正判据逻辑"。

## 四、Layer 2：在线拦截系统

### 4.1 系统定位

在线拦截是对话进行中的**实时风险分级**——在 Bot 生成回答之前，基于关键词规则判断当前 query 是否应由 AI 处理。触发红线则直接路由至人工客服，不进入 LLM 生成环节。

### 4.2 设计原则

**事前拦截优于事后补救**。事后拦截意味着 AI 已经生成了不当回答、用户已经看到，撤回体验极差且存在合规风险。事前拦截从架构层面确保高风险 query 根本不会到达 LLM。

### 4.3 规则体系

P0 规则引擎包含五条红线，按优先级串行检查：

| 优先级 | 红线       | 关键词（部分）                           | 命中动作            |
|:---:| -------- | --------------------------------- | --------------- |
| ①   | 人身安全/健康  | 宝宝、孕妇、儿童、过敏、起红点、安全、皮肤、刺激、受伤、就医    | 转人工过渡语 + 人工客服队列 |
| ②   | 赔付/退款/补发 | 退款、补发、赔偿、退差价、补偿、赔付、漏了、少发、破损、坏了、错发 | 转人工过渡语 + 售后专员队列 |
| ③   | 情绪信号强烈   | 投诉、曝光、差评、消气、离谱、坑人                 | 转人工优先队列         |
| ④   | 需联系外部方   | 物流三天、物流卡、催件、核实物流、对接快递、查询丢件、派送异常   | 转人工客服队列         |
| ⑤   | 法律边界     | 多开发票、虚开、假宣传、违规                    | 拒答 + 解释法律依据     |

**例外处理**：规则④设置意图白名单——纯物流进度查询、包邮规则咨询不触发拦截，仅"需要人工介入的物流问题"触发。

### 4.4 验证结果

**验证方式**：将 P0 规则以 Bot Prompt 内置方式（Pre-Generation）写入 v2.2 版本，在 91 条评测集上跑批验证。

| 指标    | 改前（v1 + v3.1s） | 改后（v2.2 + v3.1s） | 变化      |
| ----- |:--------------:|:----------------:|:-------:|
| D2 数量 | 19 条（20.9%）    | 0 条（0%）          | 全部拦截    |
| E 率   | 64.8%          | 97.8%            | +33.0pp |
| B 类   | 1 条            | 0 条              | 修复      |
| C 类   | 10 条           | 0 条              | 修复      |

**关键发现**：19 条 D2 case 中，②赔付退款规则触发 17 条（89%），是最集中的高风险场景。5 段 Prompt 改动实现 D2 归零，改动成本极低、效果极大。

### 4.5 生产化路径

当前验证方式为 Bot Prompt 内置（方式二），适合快速验证。生产环境推荐升级为独立中间件（方式一）：

| 维度   | 方式二：Prompt 内置（已验证）  | 方式一：中间件（推荐生产）  |
| ---- | ------------------- | -------------- |
| 实现方式 | System Prompt 嵌入规则  | Bot 服务中间件层独立实现 |
| 延迟   | 0ms（关键词匹配在 LLM 调用前） | < 5ms          |
| 可靠性  | 依赖 LLM 指令遵循能力       | 独立于 LLM，不可绕过   |
| 部署成本 | 10 分钟改 Prompt       | 需工程团队配合开发      |
| 适用阶段 | 原型验证、小团队快速上线        | 生产环境、合规要求高的场景  |

## 五、Layer 3：持续监控系统

### 5.1 系统定位

持续监控是上线后的**质量雷达**——通过定义北极星指标，持续观测 AI 客服的运行健康度，异常信号触发定向抽检，形成长期质量飞轮。

### 5.2 北极星指标体系

| 指标     | 定义                       | 异常阈值     | 触发动作         |
| ------ | ------------------------ |:--------:| ------------ |
| 转人工率   | 被规则引擎拦截的 query 占比        | 突降 > 30% | 规则可能被绕过，触发抽检 |
| 二次进线率  | 同一用户 24h 内再次咨询的比例        | 突升 > 20% | 首次回答未解决问题    |
| 客诉率    | 含"投诉/差评/曝光"关键词的 query 占比 | 突升 > 50% | 整体体验恶化       |
| AI 回答率 | Bot 实际生成回答的 query 占比     | 突升 > 20% | 规则引擎可能失效     |

### 5.3 触发式抽检流程

```
北极星指标异常
  ↓
自动拉取异常时间窗口内的对话样本（50–100 条）
  ↓
运行 Layer 1 离线评估
  ↓
定位问题根因 → 更新评测集 → 修复 Bot/规则 → 重新验证
```

### 5.4 评估 Dashboard

Dashboard 作为三层架构的统一展示层，聚合呈现：

- **整体质量指标**：E 率、D2 率、分类分布（实时/周期更新）
- **类目健康度**：五大类目的 E 率与问题分布热力图
- **拦截效果**：五条红线各自触发次数与占比
- **改前改后对比**：当前版本 vs 上一版本的指标变化
- **待修复问题清单**：按 A/B/C/D 优先级排序的迭代队列

原型阶段使用飞书多维表格手动维护 Tracing 数据；生产化后接入 BI 工具（如 Metabase/Grafana）实现数据自动同步与可视化。

## 六、实施路线图

### 6.1 原型验证阶段（已完成：1 人，4 天）

**已完成事项**：

- Coze Workflow 离线评估流水线搭建（Bot 节点 + Judge 节点）
- 91 条评测集构建 + 30 条人工标注 + Judge 三轮校准（v3.0 → v3.1 → v3.1s）
- Bot Prompt 四版迭代（v1 → v2 → v2.1 → v2.2），P0/P1/P2 方案落地验证
- Dashboard 一页纸原型 + 北极星指标定义
- **核心结论**：方法论验证通过，D2 归零，E 率 97.8%

**未在原型阶段实施（原型阶段不需要）**：

- CI/CD 集成（Coze Workflow 手动触发）
- 中间件层规则引擎（使用 Prompt 内置方式验证）
- 生产客服系统数据对接（使用飞书多维表格手动维护）

### 6.2 小团队验证（建议：1-3 人，2 周）

- Phase 1：复用现有 91 条评测集与 Judge Prompt v3.1s，根据新业务场景补充 20-30 条评测集
- Phase 2：在 Bot System Prompt 中嵌入 P0 五条红线，D2 即时归零
- Phase 3：定义人工抽检频率（建议每次发版前全量 + 每周抽检 30 条）

### 6.3 中型团队生产化（建议：5-15 人，1-2 个月）

- Layer 2 优先：规则引擎从 Prompt 方案升级为独立中间件，增加灰度发布与回滚开关
- Layer 1 自动化：评估 Pipeline 接入 CI/CD，Staging 部署后自动触发，门控不通过阻断发布
- Layer 3 上线：BI 工具接入生产客服数据，北极星指标实时监控，异常自动告警

### 6.4 大型团队增强（建议：15+ 人，持续演进）

- 评测集管理平台：在线标注 + 版本管理 + 多人协作审核
- Judge 多模型投票：2-3 个不同 LLM 独立判定，结果不一致的自动标记为"需人工仲裁"
- A/B 实验：新 Bot 版本 10% 流量 vs 旧版本 90% 流量，实时对比两组核心指标
- 对抗样本自动生成：LLM 自动改写评测集 Query（同义替换、加噪声），从 91 条扩展到 500+ 条

## 八、方法论迁移

本方案的核心方法论——**"MECE 决策树分类 + LLM-as-Judge 校准 + 规则引擎前置拦截 + 北极星指标持续监控"**——可迁移至以下企业 AI Agent 场景：

- **智能客服**（已验证）：电商、金融、政务、医疗等行业的客服 Agent 质量审计
- **知识库问答**：企业内部知识库 Agent 的回答准确性、完整性、安全边界评估
- **Agent 任务执行**：如无人机巡检 Agent、智能制造 Agent 的指令理解与安全操作边界评估
- **RAG 系统评估**：五类决策树（A 意图/B 事实/C 信息/D 安全/E 通过）天然对应 RAG Pipeline 的五个环节

迁移时需根据具体场景调整三个要素：（1）意图 Taxonomy 重构；（2）知识库内容匹配；（3）安全红线定义（不同行业的敏感场景不同）。
