# 项目中的 RAG 应用分析

> 回答：这个项目和 RAG 有什么关系？在 RAG 语境下这个项目的价值是什么？

---

## 一、两层 AI 架构

这个项目里同时运行着两套 AI 系统，扮演完全不同的角色：

```
┌──────────────────────────────────────────────────────────────┐
│                                                              │
│  Layer A：智能客服 Bot（RAG 系统）—— 被评估的对象               │
│                                                              │
│  用户 query                                                   │
│      ↓                                                       │
│  ┌──────────────┐                                            │
│  │  Retrieval   │  ← 检索 Coze 知识库（商品信息、活动规则、     │
│  │  知识库检索    │    物流FAQ、发票FAQ、售后FAQ、Bot人设）       │
│  └──────┬───────┘                                            │
│         ↓                                                    │
│  ┌──────────────┐                                            │
│  │ Augmented    │  ← 将检索到的相关文档片段拼入 LLM 上下文       │
│  │ 上下文增强    │                                            │
│  └──────┬───────┘                                            │
│         ↓                                                    │
│  ┌──────────────┐                                            │
│  │ Generation   │  ← LLM 基于增强后的上下文生成回答             │
│  │ 回答生成      │                                            │
│  └──────┬───────┘                                            │
│         ↓                                                    │
│     AI 回答                                                  │
│                                                              │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  Layer B：LLM Judge（纯 LLM 评估）—— 评估工具                   │
│                                                              │
│  输入：user_query + ai_answer（RAG Bot 的输出）                 │
│      ↓                                                       │
│  ┌──────────────┐                                            │
│  │ 决策树 Step1-4│  ← LLM 按 MECE 框架诊断回答在哪个环节出问题    │
│  │ 触发即停      │    纯 LLM 推理，不依赖外部知识库检索             │
│  └──────┬───────┘                                            │
│         ↓                                                    │
│  classification + decision_path + reason                      │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

**关键区分**：

- **Layer A（RAG）**：输入 query → 检索知识库 → LLM 增强生成 → 输出回答。RAG 在这里。
- **Layer B（Judge）**：输入 query + RAG 的输出 → LLM 推理判断 → 输出分类。这里不是 RAG——Judge 不需要检索外部知识，它需要的判据已全部内置在 prompt 中。

## 二、MECE 五类 → RAG 五环节映射

MECE 框架的五个分类天然对应 RAG Pipeline 的五个失败环节。这不是巧合——框架从"客服回答可能在哪些环节翻车"出发推导，而 RAG 的 Pipeline 恰好也是串行的。

```
RAG Pipeline：  Query ──→ Retrieval ──→ Augmented ──→ Generation ──→ Output
                   │          │             │             │            │
MECE 框架：       Step1      Step2         Step2         Step3        Step4
               意图理解    事实正确性     事实正确性     完整性与      处置策略
                           (检索质量)    (幻觉控制)    可操作性      (输出边界)
                   │          │             │             │            │
                   ▼          ▼             ▼             ▼            ▼
              A 意图错误   B 事实/断言    B 事实/断言    C 信息缺失   D 策略错误
```

### 逐环节对应

#### Step 1 → A 类：Query 理解失败（检索之前就偏了）

| RAG 环节 | 失败模式 | 项目中的 case |
|----------|----------|--------------|
| Query → Retrieval | 用户意图未被正确理解，导致检索方向错误 | I-03：用户问"发票什么时候开好"，隐含了电子和纸质两个子诉求。Bot 理解了"发票时效"但没意识到要分别回答。I-15：用户问"我这笔订单的赠品能开票吗"，Bot 理解为"赠品开票规则是什么"。 |

**RAG 视角**：这是 Retrieval 之前的失败。即使知识库有正确答案，如果 query 的理解偏了，检索出来的文档就不对，后面全白费。对应 RAG 优化方向：Query Rewrite、多意图拆分、意图分类器。

---

#### Step 2 → B 类：检索到正确信息但生成出错

B 类分两种情况，分别对应 RAG 的两个子环节：

**B-事实（检索质量问题）**

| RAG 环节 | 失败模式 | 说明 |
|----------|----------|------|
| Retrieval → Augmented | 知识库检索到的信息本身是错的，或检索到了过时的文档 | 如旧版价格、已过期的活动规则 |

**B-断言（幻觉问题）**

| RAG 环节 | 失败模式 | 说明 |
|----------|----------|------|
| Augmented → Generation | LLM 在生成时超出了检索到的知识范围，添加了知识库中没有支持的内容 | P-04："完全可以放心给宝宝用"——知识库中没有任何文档支持这个绝对化断言。LLM 在生成时自己"加戏"了。 |

**RAG 视角**：这是 RAG 最经典的失败模式——幻觉（Hallucination）。RAG 的设计初衷就是用检索来约束 LLM 的生成，但约束不是绝对的。B-断言正是"检索到了但 LLM 没遵守"的典型表现。

项目中的 91 条数据只有一个 B3（P-04），说明 Coze 的知识库检索+生成质量本身不错——幻觉控制得比较好。

---

#### Step 3 → C 类：检索到了、生成对了、但输出不完整

| RAG 环节 | 失败模式 | 项目中的 case |
|----------|----------|--------------|
| Generation → Output | 答案内容正确，但缺少让用户完成操作的关键信息 | C2 × 9 条：Bot 告诉用户"去商品详情页确认""关注秒杀专区""在订单页面申请"，但没有给链接/入口。C1 × 1 条：L-16 "专属小赠品"缺具体规则。 |

**RAG 视角**：这是 Generation → Output 环节的失败。LLM 生成了正确的内容，但知识库中的文档模板本身就不完整——FAQ 文档只有文字说明，没有嵌入 Deeplink。问题不在 LLM，在知识库文档的质量。

C 类在 v3.0 中只有 1 条，v3.1s 收紧 STAR-S 阈值后增加到 10 条——这恰好说明了 STAR-S 严格阈值对 RAG 评估的价值：它能区分"内容正确"和"内容正确且可操作"。

---

#### Step 4 → D 类：RAG 不应该回答这个问题

| RAG 环节 | 失败模式 | 项目中的 case |
|----------|----------|--------------|
| Output 边界 | RAG Pipeline 正确运行了，但这个问题本身就不应该走 RAG | D2 × 19 条：涉及赔付退款、人身安全、外部协作——AI 不应该自动回答，应该转人工。 |

**RAG 视角**：这是 RAG 架构层面最容易被忽略的问题——**不是所有 query 都应该进入 RAG Pipeline**。RAG 擅长的是"在知识库范围内回答事实性问题"，不擅长的是"做决策"（如承诺补发）、"处理安全敏感信息"（如孕妇能不能用）、"执行需要外部系统配合的操作"（如联系快递公司）。

D2 的 19 条 case 本质上不是 RAG 的质量问题，而是 **RAG 的边界问题**——需要在 RAG Pipeline 前面加一层路由器（Rule-based Router），把不适合 RAG 的问题提前拦截。

## 三、这个项目对 RAG 评估的贡献

### 3.1 评估了 RAG 的两个维度

大多数 RAG 评估只看两个指标：检索召回率（Retrieval Recall）和答案准确性（Answer Correctness）。

这个项目的 MECE 框架额外覆盖了三个通常被忽略的维度：

| 维度 | 传统 RAG 评估 | 本项目的 MECE 框架 |
|------|-------------|-------------------|
| 检索质量 | ✅ Recall@K | ✅ → B-事实 |
| 答案准确性 | ✅ BLEU/ROUGE/Correctness | ✅ → B-事实 + B-断言 |
| 意图理解 | ❌ 通常不评估 | ✅ → A 类 |
| 输出完整性 | ❌ 通常不评估 | ✅ → C 类（STAR 四维） |
| 回答边界 | ❌ 通常不评估 | ✅ → D 类（该不该答） |

### 3.2 C 类和 D 类是 RAG 评估的盲区

**C 类（输出完整性）**是 RAG 评估的典型盲区。传统评估说"答对了"——确实答对了，但用户还得自己去 App 里找操作入口。这对应 RAG 的知识库文档质量问题——文档内容正确但缺少结构化元素（链接、按钮、时效等）。

**D 类（回答边界）**更是 RAG 评估的盲区。传统评估不会问"这个回答该不该由 RAG 来生成"。但在客服场景下，一个正确的回答如果出现在不该出现的场景，就是错误。

### 3.3 评估框架本身也可以评估 RAG 组件

虽然项目当前只评估了端到端的 Bot 输出，但 MECE 框架可以进一步细化来评估 RAG 的各个组件：

```
A 类 → 评估意图分类器 + Query Rewrite
B-事实 → 评估知识库准确性 + 时效性
B-断言 → 评估 LLM 的幻觉率（检索到但没遵守）
C 类 → 评估知识库文档模板的完整性和可操作性
D 类 → 评估 Pre-RAG Router（风险分级规则）
```

## 四、Coze 平台在项目中的 RAG 角色

### 4.1 Coze 知识库 = RAG 的 Retrieval 层

项目中的 Coze 知识库包含 6 个文档：

| 文档 | 内容 | RAG 作用 |
|------|------|----------|
| 01-商品信息 | 洗衣液/洗手液/消毒液品类、规格、成分、价格 | 检索源 |
| 02-活动规则 | 满减、秒杀、会员、优惠券规则 | 检索源 |
| 03-物流FAQ | 发货时效、包邮门槛、改地址、快递公司 | 检索源 |
| 04-发票FAQ | 开票流程、专票条件、修改、冲红 | 检索源 |
| 05-售后FAQ | 退货、换货、退款时效、赔付流程 | 检索源 |
| 07-Bot人设Prompt | Bot 的语气、边界、禁止事项 | System Prompt |

当用户问"满300减多少"，Coze 的 RAG 流程是：
1. **Retrieval**：从知识库中检索与"满减"相关的文档片段（命中 02-活动规则.md）
2. **Augmented**：将检索到的片段（"满300减40，不同档位不叠加…"）拼入 LLM 的上下文
3. **Generation**：LLM 基于增强后的上下文生成回答

### 4.2 Coze Workflow = RAG 的调用层

`08-批量调用脚本.py` 做的事情就是：
1. 逐条读取 case 的 `user_query`
2. 调用 Coze API → 触发 Workflow → Workflow 内部走 RAG Pipeline → 返回 AI 回答
3. 把 AI 回答喂给 Workflow 中的 Judge 节点 → 返回分类

一条 case 在 Workflow 中经历了两次 LLM 调用：
- 第一次：RAG 检索+生成（Layer A）
- 第二次：Judge 评估（Layer B，不需要检索）

## 五、面试中的 RAG 叙事

### 如果面试官问"你做过 RAG 吗"

> "做过。这个项目的 Bot 本身就是 RAG 架构——Coze 知识库做 Retrieval，LLM 结合检索结果做 Augmented Generation。但我的重点不是搭 RAG，而是**评估 RAG**。
>
> MECE 框架的五个分类恰好对应 RAG Pipeline 的五个环节：A 类是 Query 理解（检索之前）、B 类是检索质量+幻觉控制（检索→生成）、C 类是输出完整性（生成→输出）、D 类是 RAG 的边界（哪些问题不该走 RAG）。
>
> 这个框架最有价值的地方在于评估了传统 RAG 评估不看的两件事——输出能不能直接操作（C 类的 STAR 检查），以及 RAG 有没有在不该回答的场景下越权（D 类）。"

### 如果面试官问"你觉得 RAG 最大的问题是什么"

> "从我的 91 条数据来看，RAG 最大的问题不是检索不准或幻觉——Coze 知识库质量不错，91 条里只有 1 条 B3。最大的问题是**边界**——20.9% 的 case 是 RAG 在不应回答的场景下越权作答。这说明 RAG 架构缺少一个 Pre-Retrieval Router——在进入检索之前，先判断这个问题该不该走 RAG。"

### 如果面试官问"你怎么评估一个 RAG 系统"

> "分五个维度，对应 RAG 的五个环节。意图理解（Query → Retrieval）、检索质量 + 幻觉控制（Retrieval → Generation）、输出完整性（Generation → Output）、回答边界（Output 之前的路由决策）。传统评估只看检索召回率和答案准确性，漏了意图、完整性和边界。我的 STAR 清单和三问法补的就是这三个盲区。"
