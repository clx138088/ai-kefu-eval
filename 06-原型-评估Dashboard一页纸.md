# 原型二：评估 Dashboard 一页纸

> 用途：面试时一页展示 91 条基线评估的全部核心结论  
> 数据来源：`04-Tracing表-v3.1s.md`  
> 设计原则：先总后分、红绿对照、一眼看到问题在哪

---

## 一、设计思路

### 1.1 Dashboard 的信息层级

```
第一眼（3 秒）：  整体合格率 + 最大问题
第二眼（10 秒）： 哪个类目最差 + 什么类型的错误最多
第三眼（30 秒）： 具体 case 分布 + 改进优先级
```

### 1.2 面试使用方式

不是在屏幕上放一张密密麻麻的表格。而是**边讲边指**：
- 讲"整体质量"→ 指 KPI 行
- 讲"S 类目是重灾区"→ 指类目热力图
- 讲"D2 是最大问题"→ 指 D2 分布饼图
- 讲"Judge 校准有效"→ 指 C 类演进对比

### 1.3 数据选择

用 v3.1s（严格 S 阈值）的数据，因为：
- C 类分布更接近人工标注（11.0% vs 34%，优于 v3.0 的 1.1%）
- 面试中解释 STAR-S 阈值调节本身就是一个亮点

## 三、面试讲解路线

### 路线 A：30 秒快速版

> "这是我们 91 条 baseline 的 Dashboard。一眼看过去三个数字：合格率 64.8%，但 20.9% 的 case 是 D2——AI 越权作答。S 类目是重灾区，60% 的售后 case 需要转人工但没有转。我们的 P0 方案是加一层风险分级规则引擎，预期 D2 直接归零。"

### 路线 B：2 分钟详细版

> "这个 Dashboard 总结了 91 条全量评估。先看核心 KPI——E 率 64.8%，D2 率 20.9%。D2 是什么？是 AI 在涉及赔付、人身安全、外部协作的场景下，不该自动回答但答了。最严重的是 S 类目——25 条售后投诉中 15 条判了 D2，60%。
>
> 但好消息是 D2 不需要训练更好的模型来修——89% 的 D2 触发的是同一条规则：②赔付/退款/补发。只需要在 Bot 前面加一层关键词+意图联合判断的规则引擎，命中红线直接转人工过渡语，D2 就归零了。这就是 P0 方案。
>
> 右边是我们 Judge 校准的过程——v3.0 的 C 类只有 1 条，和人工标注差 33 个百分点。经过 STAR 清单和三问法两轮迭代，C 类提升到 11%。这个 Dashboard 用的是 v3.1s 的数据，严格 S 阈值版。"

## 四、Dashboard 设计原则复盘

| 原则 | 体现 |
|------|------|
| **一眼见好坏** | KPI 行三个大数字 + 颜色编码（绿✅/红🔴/黄📋） |
| **一眼见问题在哪** | 类目热力图——S 行全是红，A 行全是绿 |
| **一眼见怎么修** | 每个失败模式直接带 P0/P1/P2 方案 |
| **展示校准过程** | 三版 Judge 对比表——证明框架不是拍脑袋的 |
| **可追溯** | 所有数字来自 Tracing 表，每一条都有 decision_path |
| **面试友好** | 不依赖屏幕——打印出来拿在手上指，或平板横屏展示 |

## 五、如果面试官追问

**"64.8% 的 E 率，你觉得能上线吗？"**

> 不能。基线 E 率只是一个快照——20.9% 的 D2 意味着每 5 次售后对话就有 3 次 AI 越权。但好消息是修复成本很低。P0 规则引擎加上后，D2 归零，E 率会升到 85% 以上。到那时才具备上线条件。

**"为什么用 v3.1s 而不是 v3.0 的数据？"**

> v3.0 的 C 类只有 1 条，明显偏低——LLM 太宽容了。v3.1s 收紧 S 阈值后 C 类 11.0%，和人工标注的 34% 差距缩小到 23pp。这 23pp 的差距可以用 Coze Bot 回答质量更好来解释。v3.1s 是当前最接近真实的快照。

**"S 类目 60% D2，是不是说明 Bot 完全不能处理售后？"**

> 不是。25 条 S case 中有 10 条判了 E——说明 FAQ 型售后（如"退款什么时候到账""怎么退货""换货运费谁出"）Bot 处理得很好。D2 集中在涉及赔付承诺和外部协作的场景。所以不是"售后不能 AI 化"，而是"售后的某些子场景需要转人工"。规则引擎正好能做这个细分。
