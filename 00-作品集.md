# 作品集：AI 智能客服评估体系

> 作者角色：AI 产品经理（评估框架设计 + Judge Prompt 工程 + 数据分析）  
> 场景：蓝月亮电商客服（洗衣液/洗手液等日化品类）  
> 周期：2026-06-05 ~ 2026-06-12  
> 核心命题：**如何系统性地评估一个 AI 客服能不能上线？**

## 模块一：问题定义

### `01-评测集-91条.md`

**这份文档回答**：你评估的 Bot 面对的是什么？

- 5 个类目（商品咨询 P / 活动规则 A / 物流查询 L / 发票开具 I / 售后投诉 S）× 17 个子场景
- 91 条 query 覆盖不同难度：单意图 FAQ → 多意图复合 → 边界 case
- 每条 case 标注了人工预期意图
- 答案覆盖率 100%（从原始 28.6% 补齐至全量）

**作品集看点**：评测集不是随便写 91 句话——每条 query 都对应一个具体的意图节点，确保了类目×子场景的矩阵覆盖。

---

## 模块二：评估框架

### `02-MECE评分标准.md`

**这份文档回答**：凭什么你的评估标准是靠谱的？

- 从"一个客服回答可能在哪些环节翻车"出发，列出 9 种失败模式
- 合并为 A/B/C/D/E 五类互斥决策树（MECE 验证通过）
- 每个分类有明确的触发条件和停止规则（Step 1→2→3→4，触发即停）
- 含 6 次边界 case 追问的推导过程

**作品集看点**：不是直接用现成的打分框架——从场景出发推导自己的标准，且能说清楚每一步为什么这么分。

### `03-Judge-Prompt-v3.1s.md`

**这份文档回答**：怎么让 LLM 帮你自动评估，而不是人工标注 91 条？

- 单 prompt 替代四节点并行评分
- 决策树嵌入 prompt：Step 1→2→3→4，触发即停
- STAR 四维完整性检查（S步骤/T时效/A替代/R参照）
- 三问法 D2 边界判断（Q1谁在承诺 Q2转人工前给了方案 Q3AI自称"我"）
- 从 v3.0→v3.1→v3.1s 的三版校准链路

**作品集看点**：Judge Prompt 工程是这份作品集中最有 AI PM 辨识度的部分——展示的不是"写了一个 prompt"，而是"如何系统性地校准一个 prompt"。

## 模块三：评估执行与数据

### `04-Tracing表-v3.1s.md`

**这份文档回答**：91 条 case 逐一判下来，每一条的结论和依据是什么？

- 91 条全量分类数据，每条含：显性诉求、隐含诉求、分类、等级、decision_path
- v3.1s 基准（严格 S 阈值），标注了与 v3.1 的 9 条改判
- 分类统计：E 59 / D2 19 / C2 9 / A1 1 / A2 1 / B3 1 / C1 1

**作品集看点**：每一条都有 decision_path——可以追溯到具体在哪个 Step、因为什么条件触发。这不是"LLM 觉得 3 分"，而是"Step 4 Q1+Q2 触发 → D2"。

### `00-评测结果迭代报告.md`

**这份文档回答**：三个版本的 Judge 跑下来，哪些结论是稳定的，哪些是校准敏感的？

- 三版数据对比表：A/B/D 三版一致（稳定），C 类随 S 阈值变化（校准敏感）
- 稳定结论：D2 19 条（89% 触发同一条规则）、S 类目 60% D2 率、A 类目 100% E 率
- 最终 E 率路径：64.8% →（P0）→ 85.7% →（P1）→ 95.6% →（P2）→ 100%

**作品集看点**：区分"Judge 校准导致的变化"和"Bot 质量的真实发现"——这是数据驱动产品的核心素养。

## 模块四：结论与洞察

### `05-基线评估报告-v3.1s.md`

**这份文档回答**：所以这个 Bot 到底能不能上线？为什么？怎么改？

- 分类分布 + 按类目交叉分析
- Top 3 失败模式分析（D2 越权作答 / C2 入口缺失 / A+B3+C1 内容缺陷）
- P0/P1/P2 迭代方案（含覆盖范围、预期收益、改动成本、验证方式）
- 三版本框架对比 + 版本选择建议

**作品集看点**：评估不是为了证明 Bot 行不行，是为了告诉团队**改哪里、先改什么、改了能好多少**。

## 模块五：产品方案

### `06-原型-P0转人工规则引擎.md`

**这份文档回答**：怎么把评估发现变成产品方案？

- Mermaid 流程图：意图分类→五条红线串行检查→高风险转人工/低风险Bot生成→三问法自检兜底
- 规则定义：五条红线（人身安全/赔付退款/情绪信号/外部协作/法律边界）
- 19 条 D2 逐条标注会被哪条规则拦截
- 规则引擎伪代码（Python）
- 预期效果：D2 19→0，E 率→85.7%

**作品集看点**：不是画个流程图就完了——每条 D2 case 都能在规则引擎中找到归宿。方案是可以被逐条验证的。

### `06-原型-评估Dashboard一页纸.md`

**这份文档回答**：怎么把 91 条数据变成一眼能看懂的东西？

- 信息层级 3 秒→10 秒→30 秒递进设计
- ASCII 可视化：KPI 行 + 类目热力图 + D2 分布 + C 类演进
- 每个失败模式直接带 P0/P1/P2 方案
- Dashboard 设计原则复盘

**作品集看点**：数据产品思维——不只是"展示数据"，而是设计信息层级让不同角色（老板/PM/工程师）各取所需。

## 模块六：方法论反思

### `00-RAG应用分析.md`

**这份文档回答**：这个项目和 RAG 有什么关系？技术深度体现在哪？

- 两层 AI 架构：Layer A（RAG Bot）被 Layer B（LLM Judge）评估
- MECE 五类 → RAG Pipeline 五环节映射：A=Query理解、B=检索+幻觉、C=输出完整性、D=RAG边界
- C 类和 D 类是传统 RAG 评估（只看 Recall+Correctness）的两大盲区

**作品集看点**：不是"搭了一个 RAG"，而是"评估了一个 RAG"。而且评估框架本身能拆开评估 RAG 的每个组件。

### `00-业务落地应用方案.md`

**这份文档回答**：这套评估体系在实际业务中怎么运转起来？

- 三层架构：Layer 1 离线评估（质量门控）→ Layer 2 在线拦截（实时决策）→ Layer 3 持续监控（异常发现）
- 完整业务闭环图
- 三种规模团队（小/中/大）的差异化落地路径
- 关键风险与应对

**作品集看点**：不是"做完评估就完了"——有一套从离线到在线、从发版前到上线后的完整质量保障闭环。
