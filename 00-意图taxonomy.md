# 智能客服意图分类体系

> 业务背景：蓝月亮电商客服（官方商城 / 电商平台旗舰店）  
> 用途：智能客服评估的意图路由参考标准  
> 配套：`01-评测集-91条.md`（每条 case 标注了预期意图）

---

## 一、分类总览

```
Level 1：业务线
  └── 电商零售（蓝月亮官方商城 / 电商平台旗舰店）
        └── Level 2：意图大类 × 5
              ├── 商品咨询（P）
              ├── 活动规则（A）
              ├── 物流查询（L）
              ├── 发票开具（I）
              └── 售后投诉（S）
              └── Level 3：具体场景 × 每类 4-7 个
```

## 二、商品咨询（P）

**定义**：用户对商品本身信息（规格、价格、库存、使用方法、适用场景）的主动查询。

**非该意图**：订单中的商品状态 → 物流查询。商品质量问题 → 售后投诉。

| 场景编号  | 场景名称    | 典型 query            | 自动化难度 | 备注                       |
| ----- | ------- | ------------------- | ----- | ------------------------ |
| P-01  | 商品规格/成分 | "洗衣液是多少ml的""含不含荧光剂" | 🟢 低  | 知识库有标准答案                 |
| P-02  | 价格/优惠叠加 | "这个现在多少钱""能用优惠券吗"   | 🟡 中  | 价格动态变化；见 P-02 边界判例       |
| P-03  | 库存/补货   | "XX款还有货吗""什么时候补货"   | 🟡 中  | 库存实时性；缺货应推荐替代品           |
| P-04a | 使用方法/用量 | "这个怎么用""一次用多少"      | 🟢 低  | 标准 SOP 可覆盖               |
| P-04b | 特殊人群适用  | "宝宝能用吗""孕妇能用吗"      | 🟡 中  | 涉及人身安全，默认推荐专用系列 + 建议咨询人工 |
| P-05  | 商品对比    | "这个和XX有什么区别"        | 🟡 中  | 需理解差异点；知识库应预置对比型 FAQ     |

### P-02 边界判例（价格 vs 活动 vs 多意图）

P-02 是商品咨询和活动规则之间最高频的模糊地带。

**判据——"能不能只靠一个信息源回答？"**

```
能，且只需要商品库 → 商品咨询
能，且只需要活动配置 → 活动规则
不能，至少需要两个信息源 → 多意图
```

| query         | 归类                | 判据关键          |
| ------------- | ----------------- | ------------- |
| "这个洗衣液多少钱"    | P-02 价格           | 信息源=商品库       |
| "满300减多少"     | A-01a 满减          | 信息源=活动配置      |
| "到手多少钱"       | 多意图（P-02 + A-01a） | 需商品库+活动库+计算   |
| "有没有参与哪些优惠活动" | A-01b 适用商品        | 商品是查询条件而非查询目标 |

## 三、活动规则（A）

**定义**：用户对平台/店铺促销活动规则、参与方式、适用范围的查询。

**非该意图**：活动订单的物流问题 → 物流查询。活动商品质量问题 → 售后投诉。

| 场景编号  | 场景名称      | 典型 query             | 自动化难度 | 备注                |
| ----- | --------- | -------------------- | ----- | ----------------- |
| A-01a | 满减/满赠规则本身 | "满300减多少""买几送几"      | 🟢 低  | 规则固定，FAQ 直接答      |
| A-01b | 活动适用商品查询  | "哪些商品参加满减""这个参加什么活动" | 🟡 中  | 需查活动-商品映射表        |
| A-02  | 优惠券使用     | "优惠券在哪领""为什么用不了"     | 🟡 中  | 多条件判断；查券状态需对接账户系统 |
| A-03  | 限时/秒杀     | "秒杀几点开始""还有名额吗"      | 🟡 中  | 时效性强；应加设置提醒引导     |
| A-04  | 赠品规则      | "买洗衣液送什么""赠品能选吗"     | 🟢 低  | 规则固定              |
| A-05  | 会员专享      | "会员有什么优惠""怎么升级"      | 🟡 中  | 会员等级差异；查等级需对接账户系统 |

### A-01 边界判例（活动规则 vs 物流查询 vs 商品咨询）

```
"满300减40的规则是什么？"          → A-01a（信息源：活动配置）
"哪些商品参加满300减40？"          → A-01b（信息源：活动-商品映射表）
"这个商品参加什么活动？"            → A-01b（信息源：商品-活动映射表）
"这个参加满减后到手多少钱？"        → 多意图（商品价格 + 活动规则 + 计算）
"活动订单什么时候发货？"（问规则）   → A-01a（信息源：活动配置）
"活动订单什么时候发货？"（问我的单） → L-02（信息源：具体订单）
```

## 四、物流查询（L）

**定义**：用户对订单物流状态、配送时效、运费、收货地址修改的查询。

**非该意图**：物流导致的售后问题（用户已归因）→ 售后投诉。发票物流 → 发票开具。

| 场景编号  | 场景名称    | 典型 query            | 自动化难度   | 备注                   |
| ----- | ------- | ------------------- | ------- | -------------------- |
| L-01  | 物流进度    | "我的快递到哪了"           | 🟢 低    | API 查询，返回轨迹+预计送达     |
| L-02  | 配送时效    | "什么时候能送到""能加急吗"     | 🟢 低    | 规则明确；大促期间应加延迟提示      |
| L-03  | 运费/包邮   | "运费多少""满多少包邮"       | 🟢 低    | 规则明确；应告知差额和凑单引导      |
| L-04a | 修改地址咨询  | "怎么改地址""在哪改"        | 🟢 低    | 纯规则查询                |
| L-04b | 状态依赖改址  | "已发货还能改地址吗"         | 🟡 中    | 需查物流状态；省内→告知方案；跨省→升级 |
| L-04c | 拦截/跨省改址 | "已发货能改到XX省吗"        | 🔴 高    | 需拦截快递，成本高+需人工协调      |
| L-05  | 物流异常    | "物流三天没更新""显示签收但没收到" | 🟡 中→🔴 | 见 L-05 边界判例          |

### L-05 边界判例（物流异常 vs 售后投诉）

**核心判据——用户是否已经完成了自我归因？**

```
未归因（只说现象）→ 先走物流查询，给信息确定性，末尾埋转售后钩子
已归因（定性为谁的错）→ 直接售后投诉
```

| query         | 归因状态     | 归类         | 处置                 |
| ------------- | -------- | ---------- | ------------------ |
| "物流三天没更新了"    | 未归因      | L-05 物流异常  | 查物流 + 催件 + 末尾挂售后钩子 |
| "怎么还没到"       | 未归因      | L-05 物流异常  | 同上                 |
| "快递显示签收但我没收到" | 未归因但高风险  | L-05       | 先引导排查，无果转售后        |
| "承诺送货上门却放驿站"  | 已归因（违约）  | S-06b 售后投诉 | 直接致歉+补救+转人工        |
| "你们说话不算话"     | 已归因+道德指控 | 售后投诉       | 高优先级转人工            |

## 五、发票开具（I）

**定义**：用户对发票开具流程、类型、状态、修改的查询。

**非该意图**：发票相关的售后投诉 → 售后投诉。

| 场景编号 | 场景名称 | 典型 query           | 自动化难度 | 备注                  |
| ---- | ---- | ------------------ | ----- | ------------------- |
| I-01 | 申请开票 | "怎么开发票""在哪登记"      | 🟢 低  | 引导自助操作              |
| I-02 | 发票类型 | "能开专票吗""个人能开企业抬头吗" | 🟢 低  | 规则明确；专票需一般纳税人认证     |
| I-03 | 发票进度 | "发票什么时候开好""审核要多久"  | 🟢 低  | 通用时效；查具体状态需对接系统     |
| I-04 | 发票修改 | "抬头写错了能改吗""能重开吗"   | 🟡 中  | 需判断发票状态；涉及作废重开应转人工  |
| I-05 | 发票内容 | "能开办公用品吗""金额能多开吗"  | 🟡 中  | 合规限制；违规请求应拒答+说明法律依据 |

## 六、售后投诉（S）

**定义**：用户因商品质量、物流破损、漏发错发、服务履约、退款等问题提出的投诉或售后诉求。

**非该意图**：仅查询售后进度但未归因 → 仍属此类。仅表达不满但不要求处理 → 仍属此类（低优先级）。

| 场景编号  | 场景名称   | 典型 query                  | 自动化难度 | 备注                |
| ----- | ------ | ------------------------- | ----- | ----------------- |
| S-01  | 质量问题   | "洗衣液漏了""味道不对""手上起红点了"     | 🔴 高  | 涉及产品安全/人身健康，必须转人工 |
| S-02  | 漏发/错发  | "少发了一瓶""收到的不是我买的"         | 🔴 高  | 需核实仓库+补发/退款       |
| S-03a | 退款时效咨询 | "退款要多久""什么时候能退"           | 🟢 低  | 通用规则：3-7个工作日      |
| S-03b | 退款状态查询 | "我的退款到哪了""为什么还没到"         | 🟡 中  | 需对接支付/财务系统        |
| S-03c | 退款账户查询 | "退款退到哪个账户"                | 🟡 中  | 需查支付方式；账户异常→升级    |
| S-03d | 退款被拒申诉 | "为什么退款被拒绝了"               | 🔴 高  | 权益纠纷+情绪升级风险       |
| S-04a | 退货流程咨询 | "怎么退货""退货要什么材料"           | 🟢 低  | 纯规则查询             |
| S-04b | 状态依赖退货 | "已发货还能退吗""用过还能退吗"         | 🟡 中  | 需查订单状态+匹配规则       |
| S-04c | 退货纠纷   | "你们拒绝了我的退货申请"             | 🔴 高  | 争议判定+用户情绪         |
| S-05  | 破损/丢件  | "箱子烂了""快递说送到但我没收到"        | 🔴 高  | 需拍照核实+赔付/补发       |
| S-06a | 换货流程   | "换货运费谁出"                  | 🟢 低  | 流程明确；质量问题→商家承担    |
| S-06b | 服务履约纠纷 | "承诺送货上门却放驿站""说了不要圆通还是发圆通" | 🟡→🔴 | 情绪信号改写意图分类，见判例    |

### S-06b 判例（情绪信号改写意图分类）

**核心判据——情绪信号是否足以让 query 从"查询"升级为"投诉"**

```
物流未按时配送 vs 承诺送货上门却放驿站：

                物流未按时配送              承诺送货上门却放驿站
用户知道货在哪    不确定                      知道（驿站）
用户要什么        信息 + 催促                 道歉 + 补救 + 补偿
情绪信号          可能有，较温和              明确（"承诺"+"却"+"未经同意"）
用户归因状态      未归因                      已归因（定性为违约）
归类              物流查询 → 售后钩子          直接售后投诉
系统第一步        查物流状态                  致歉 + 确认诉求 + 转人工
```

**情绪信号关键词**：承诺/说好的、却/竟然/结果、未经同意/直接/擅自、说话不算话/骗人、投诉/曝光/差评。

## 七、联合意图规则

### 优先级（解决"先处理哪个"）

```
售后投诉 > 物流查询（涉及赔付的物流问题先按售后处理）
售后投诉 > 发票（先处理售后，发票通常是售后的前置条件）
```

### 并行规则（解决"能不能同时处理"）

```
活动规则 + 商品咨询 → 并行处理，各自回答
物流 + 发票 → 并行处理，无依赖
```

### 多意图处置原则

```
不是所有 query 都需要精准归属到一个且仅一个类目。

query → 多级判断 → 主类目 + 置信度 + 潜伏类目
     → 主链路先跑
     → 答案末尾根据潜伏类目埋钩子
     → 用户点钩子 → 切到副链路

例："大促订单什么时候发货"
  主类目=活动规则（问时效规则）
  潜伏类目=物流查询（用户可能接下来问我的订单）
  回答："大促期间发货时效为下单后72小时内。如需查询您的订单状态→[查物流]"
```

### 高频联合意图

| 联合意图         | 容易误判为   | 正确做法                  |
| ------------ | ------- | --------------------- |
| 活动规则 + 物流    | 仅活动规则   | 分别回答满减规则和发货时间         |
| 商品 + 发票      | 仅商品     | 回答完商品后引导开票            |
| 物流 + 售后      | 仅物流     | 未归因→先查物流+埋钩子；已归因→直接售后 |
| 活动 + 商品（到手价） | 仅商品或仅活动 | 并行：标价 + 活动规则 + 到手价估算  |
| 活动 + 物流 + 发票 | 仅第一个匹配  | 逐一回答所有子意图             |

## 八、自动化难度决策树

**不按类目统一标难度。按以下决策树逐条判断：**

```
Step 1：回答这条 query 是否需要查询用户个人的实时数据？
        （订单状态/物流轨迹/支付方式/优惠券资格/会员等级）
  ├─ 不需要 → 🟢 低
  └─ 需要 → Step 2

Step 2：查到数据后，是否涉及以下任一高风险因素？
        · 赔付金额/退款
        · 人身安全/健康
        · 法律合规（发票虚开等）
        · 快递拦截/跨省改址
        · 用户情绪已升级为投诉
  ├─ 不涉及 → 🟡 中
  └─ 涉及 → 🔴 高，必须转人工
```

### 情绪信号对难度的修正

当 query 包含以下信号时，难度至少上调一级（🟢→🟡，🟡→🔴）：

- "承诺"/"说好的" → 用户有契约锚点
- "却"/"竟然"/"结果" → 意外+不满
- "未经同意"/"直接"/"擅自" → 被侵犯感
- "说话不算话"/"骗人" → 道德指控
- "投诉"/"曝光"/"差评" → 明确威胁

**原因**：即使问题本身是标准 FAQ 可覆盖的，带情绪信号的 query 若被 AI 机械回答极易激化矛盾。此时 AI 的任务不是给正确答案，而是先接住情绪再转人工。
