## 三、稳定结论（三版一致的发现）

以下结论不随 Judge 版本变化——是 Bot 质量的确定性快照。

### 3.1 D2 是最大问题（19 条，稳定于三版）

| 触发条件       | 数量  | 占比  | 代表性 case                        |
| ---------- | --- | --- | ------------------------------- |
| ② 赔付/退款/补发 | 17  | 89% | S-01~S-22（售后类）                  |
| ④ 需联系外部方   | 15  | 79% | L-05/L-11/L-15, S-01/S-05/S-10等 |
| ① 人身安全/健康  | 2   | 11% | P-11, S-06                      |
| ③ 情绪信号强烈   | 1   | 5%  | S-14（"我要投诉"）                    |

> 89% 的 D2 触发同一条规则（②赔付退款），说明问题高度集中，规则引擎是最优解。

### 3.2 S 类目是重灾区（60% D2 率，稳定于三版）

| 类目     | D2 数 | D2 率  | 三版一致性 |
| ------ | ---- | ----- | ----- |
| S 售后投诉 | 15   | 60.0% | ✅     |
| L 物流查询 | 3    | 18.8% | ✅     |
| P 商品咨询 | 1    | 5.9%  | ✅     |
| A 活动规则 | 0    | 0%    | ✅     |
| I 发票开具 | 0    | 0%    | ✅     |

### 3.3 A 类目是标杆（100% E 率，稳定于三版）

活动规则类 16 条全部为 E——FAQ 型、规则明确、无安全风险、不需外部方——是 Bot 最擅长的场景。

### 3.4 少量内容质量缺陷（稳定于三版）

| case | 分类  | 问题                     |
| ---- | --- | ---------------------- |
| P-04 | B3  | "完全可以放心给宝宝用"——替品牌做安全承诺 |

## 四、校准敏感结论（随版本变化的发现）

### 4.1 C2 从 0 → 9 条（v3.1s 新增）

v3.1s 收紧 STAR-S 阈值后，9 条 case 从 E 降级为 C2。这 9 条不是 Bot 的内容有错，而是**用户需要去 App 操作但 Bot 没给直达入口**。

| 类目     | C2 数 | 典型缺失                                       |
| ------ | ---- | ------------------------------------------ |
| A 活动规则 | 4    | A-04 秒杀入口、A-05 赠品确认入口、A-11 加购链接、A-14 优惠券入口 |
| L 物流查询 | 2    | L-04 改地址入口、L-09 跨省改地址入口                    |
| P 商品咨询 | 1    | P-16 成分表链接                                 |
| I 发票开具 | 1    | I-16 自助开票链接（提到但未给）                         |
| S 售后投诉 | 1    | S-12 退差价入口                                 |

### 4.2 C 类检测率与人工基准的差距

| 版本    | C 类率  | 与人工(34%)差距 | 说明                   |
| ----- | ----- | ---------- | -------------------- |
| v3.0  | 1.1%  | -33pp      | LLM 严重"脑补"缺失信息       |
| v3.1  | 1.1%  | -33pp      | STAR 框架正确但S阈值过松      |
| v3.1s | 11.0% | **-23pp**  | 差距缩小，剩余可由 Bot 质量优势解释 |

> 剩余 23pp 差距的合理解释：人工标注时使用的是"模拟关键词匹配 Bot"的回答，而 LLM 评估的是 Coze Bot 的真实输出。Coze Bot 的回答确实更完整——这是 Bot 质量的真实提升，不是 Judge 的错误。

> **Coze Bot 在电商客服场景下，FAQ 型 query（活动规则、发票、商品信息）回答质量较高。核心短板在售后场景——涉及赔付、人身安全、外部协作时，Bot 应转人工但普遍越权作答。**

## 七、迭代报告方法论总结

### 7.1 本次迭代的特殊性

这次迭代不是通常意义上的"Bot 改了一版、重跑看效果"，而是 **Judge 本身在迭代**——评估工具和被评估对象同时在被校准。这要求我们：

1. **区分"稳定结论"和"校准敏感结论"**：A/B/D 三版一致 → 稳定；C 类随 S 阈值变化 → 校准敏感
2. **标注每次改动的驱动因素和效果**：不是为了改而改，每次都有数据驱动的理由
3. **保留多版本可回溯**：如果未来有人质疑"为什么 C 类是 11% 不是 1%"，可以追溯到 v3.0→v3.1→v3.1s 的完整校准链路

### 7.2 后续迭代的正常流程

当 Judge 稳定后（v3.1s 作为当前基准），后续 Bot 迭代的评估流程变为：

```
Bot v2 → 同一评测集 + 同一 Judge(v3.1s) → 新 Tracing 表 → 对比 v1 基线 → 迭代报告
```

此时 Judge 不再变化，所有变化来自 Bot 本身——这才是"正常的"迭代评估。
