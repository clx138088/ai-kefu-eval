# AI 智能客服评估项目总结

> 项目周期：2026-06-05 ~ 2026-06-11  
> 场景：蓝月亮电商客服（洗衣液/洗手液等日化品类）  
> 角色：评估框架设计者 + Judge Prompt 工程者 + 数据分析者

---

## 一、项目定位

### 一句话

**用 MECE 决策树替代主观打分，构建可复现、可审计的 AI 客服回答质量评估体系。**

### 解决什么问题

智能客服 Bot 上线前需要回答一个核心问题：**这 Bot 能上线吗？** 传统的"人工打分"有三个硬伤：

| 硬伤 | 表现 | 后果 |
|------|------|------|
| 总分掩盖根因 | "这个回答 3 分"——不知道扣分是因为理解错了、说错了、还是该转没转 | 无法定位改进方向 |
| 维度不独立 | 意图理解错误导致下游信息缺失，两个维度都扣分 | 同一错误被双重惩罚 |
| 量纲不同等权加和 | "意图理解"和"操作指引"是不同的东西，但都是 1-5 分等权相加 | 分数不可比 |

**这个项目做的是**：不评分，诊断。像医生不告诉你"健康分 70 分"，而是告诉你"肺部有问题、心脏没问题、血压偏高"。

### 项目在 AI 产品面试中的定位

这是 AI 产品的**评估体系建设**能力证明。它展示的不是"我训练了一个模型"，而是：

- 我知道 AI 产品上线前需要评估什么
- 我能设计评估框架（不是直接用人家的开源 benchmark）
- 我能校准评估工具（Judge Prompt 工程是 AI PM 的核心技能）
- 我能从评估结果推导产品迭代方案

---

## 二、开展思路

### 核心设计原则

```
不要从"怎么打分"出发 → 从"一个客服回答可能在哪些环节翻车"出发
```

推导过程：

```
一个客服回答的"失败"可能发生在：
  1. 没听懂用户问什么 → 意图理解失败
  2. 听懂了但说错了 → 事实错误或胡说
  3. 听懂了也说对了但没说完 → 信息缺失
  4. 听懂说对说全了但不该由AI来说 → 处置策略错误
  
这四个环节是互斥的、有因果顺序的 → 成为 MECE 决策树的四个 Step
```

### MECE 五类框架

| 类别 | 含义 | 决策树位置 | 对应改进方向 |
|------|------|-----------|-------------|
| E | 无问题 | 全部通过 | — |
| A | 意图理解错误 | Step 1 触发 | 意图分类器 / Query Rewrite |
| B | 事实错误 / 断言无据 | Step 2 触发 | 知识库 / 幻觉控制 |
| C | 信息缺失 / 不可操作 | Step 3 触发 | 答案模板 / 结构化引用 |
| D | 处置策略错误 | Step 4 触发 | 风险分级 / 转人工规则 |

**MECE 验证**：五类相互排斥（一条 case 只能触达一个停止点）、集体穷尽（一个回答的所有可能问题都被覆盖）。

### 阶段一：框架搭建 + 评测集构建（Day 0-2）

**目的**：建立"评什么"和"怎么评"的基础。

**做了什么**：
- 从蓝月亮电商客服场景出发，构建意图 taxonomy（5 大类目 × 多级子场景）
- 手工编写 50 条评测 query，覆盖 P商品/A活动/L物流/I发票/S售后 五个类目
- CSV 补齐至 91 条，**有答案覆盖率从 28.6%（26/91）提升至 100%（91/91）**
- 10 条人工标注训练（初检 71.1 → 复检 85.3 一致性）

**产出**：`00-意图taxonomy.md`、`01-评测集-91条.md`

**AI 相关性**：意图 taxonomy 的自动化难度决策树——判断哪些意图适合 AI 自动处理、哪些必须转人工。这是 AI 产品的需求分析：不是所有用户问题都应该由 AI 回答。

---

### 阶段二：MECE 评分标准推导（Day 3）

**目的**：从零推导一套互斥且穷尽的评估框架。

**做了什么**：
- 诊断旧版四节点并行评分框架的三个硬伤（总分算错、维度不独立、量纲不同等权加和）
- 从"一个客服回答可能在哪些地方翻车"出发，列出 9 种失败模式
- 合并同类项、切分维度、反复迭代边界 case，最终收敛到 A/B/C/D/E 五类决策树
- 六次边界 case 追问完善判据（P-02/A-01/L-05/S-06b 等）

**核心推导案例**：

| case | 旧判 | 新判 | 为什么改 |
|------|------|------|----------|
| P-09 | B3（事实错误） | A3（意图理解） | 问"能不能洗羊毛衫"答"怎么洗"——不是答错了，是根本没答到点上 |
| P-02 | C2（信息缺失） | A1（意图理解） | 答了区别但没给出选择建议——隐含诉求未被捕捉 |
| P-07 | C2（信息缺失） | A3（意图理解） | 问"何时补货"答"无货"——单一显性诉求回应错 |
| S-01 | E（无问题） | D2（策略错误） | AI 在转人工前承诺了赔付方案——回答内容对了但策略错了 |

**产出**：`02-MECE评分标准.md`

**AI 相关性**：MECE 决策树的 B 类专门区分了"事实错误"和"断言无据"——后者是 LLM 幻觉的典型表现（"绝对安全""完全可以放心"）。将 AI 特有的失败模式纳入评估框架，体现了对 LLM 能力的理解。

---

### 阶段三：Judge Prompt 工程 + Workflow 设计（Day 3-4）

**目的**：将 MECE 框架转化为 LLM 可执行的评估指令，实现规模化自动评估。

**做了什么**：
- 编写单 prompt Judge（替代旧版四节点并行 prompt）
- 设计触发即停的决策树执行逻辑
- 发现"概括描述"不触发 Step 2 的问题，新增前置检查步骤
- P-05/P-04 试跑纠错，校准 judge 判断
- 设计三个 Mermaid 流程图：单条评估流、全量项目流、分类→链路→优化映射
- 在 Coze 平台搭建模拟客服 Bot，接入知识库，批量调用 API 跑 91 条数据

**产出**：`03-Judge-Prompt+Workflow.md`（v3.0）

**AI 相关性**：
- **Judge Prompt 工程**是 AI PM 的核心技能——如何让 LLM 作为评估者输出稳定、一致的结果。设计中的"前置检查（概括描述 vs 逐字原文）""触发即停""锚点内置到每个分支""输出带 decision_path"都是 prompt engineering 的具体实践。
- **用 AI 评估 AI** 是本项目的核心 AI 叙事——不是人工标注几百条数据，而是用 LLM 做规模化质量审计。

---

### 阶段四：基线数据跑批 + 报告生成（Day 4）

**目的**：获取 91 条全量评估数据，输出基线结论。

**做了什么**：
- Coze Workflow 批量跑 Judge Prompt，获得 91 条全量分类
- 填写 `04-Tracing表.md`
- 统计分类分布、按类目交叉分析、识别 Top 3 失败模式
- 制定 P0/P1/P2 迭代方案
- 编写面试防御（5 个常见追问的回答）

**核心发现**：

| 指标 | 数值 |
|------|------|
| AI 回答合格率（E） | 74.7% |
| 该转未转率（D2） | 20.9% |
| S 类目 D2 率 | **60%**（重灾区） |
| A 类目 E 率 | **100%**（标杆） |

**产出**：`04-Tracing表.md`、`05-基线评估报告.md`

---

### 阶段五：Judge 校准迭代（Day 5 / 6月11日）

**目的**：通过与人工标注对比，发现并修复 LLM Judge 的系统性偏差。

**做了什么**：
- 将 91 条 LLM 结果与 50 条人工 MECE v3.0 重分类做逐项对比
- 发现三个系统性偏差：C 类检测过弱、D2 判据④模糊、B/D 双触发未记录
- 进行三轮迭代修复：

```
v3.0 → v3.1（大改）           v3.1 → v3.1s（小改）
──────┼──────────────────    ──────┼──────────────────
      │                            │
  Step 3: 抽象问题→STAR清单       STAR-S阈值: 宽松→严格
  Step 4: 一句话→三问法+10锚点     "有路径描述"→"需链接或≥4步导航"
  Step 2: +B/D双触发规则
  Step 1: +锚点case
      │                            │
  0改判，推理质量提升             9条改判(E→C2)，C类从1.1%→11.0%
```

**产出**：
- `03-Judge-Prompt-v3.1.md`、`03-Judge-Prompt-v3.1s.md`
- `04-Tracing表-v3.1.md`、`04-Tracing表-v3.1s.md`
- `05-基线评估报告-v3.1.md`、`05-基线评估报告-v3.1s.md`
- `Coze素材/人工vsLLM-Judge差异分析.md`
- `Coze素材/v3.1验证-10条case重评对比.md`
- `参考资料/Judge-Prompt三版本对比分析.md`

**AI 相关性**：
- **Judge 校准**是 AI 评估体系中最容易被忽略但最重要的环节。类比 ML 中的模型评估——你需要知道你的评估器的准确率、召回率、一致性。这个环节展示了"不信任 LLM 的输出，而是把它当作一个需要校准的测量仪器"的工程思维。
- **三问法**是 prompt engineering 解决 LLM 判断不一致问题的具体案例——将模糊的自然语言判据（"是否涉及联系外部方"）拆解为三个可逐一回答的条件判断（"谁在承诺？转人工前给了方案？AI 自称'我'？"）。

## 五、项目亮点

### 1. 框架原创性

不是套用现成的评估 benchmark（如 BLEU/ROUGE 之于翻译，或直接用 GPT-4 打分），而是从电商客服场景的实际失败模式出发，从头推导 MECE 分类框架。A/B/C/D/E 五类的边界经过了六次 case 追问和 50 条人工标注的检验。

### 2. Judge Prompt 工程深度

Judge Prompt 经历了三个版本的迭代，每个版本的改动都有明确的驱动因素（数据偏差）、具体的改动点（STAR/三问法/阈值收紧）和可量化的效果（C 类从 1.1% 到 11.0%）。这不是"调了几次 prompt"，而是系统性的校准流程。

### 3. 用 AI 评估 AI

不是人工标注 91 条 × 10 分钟 = 15 小时，而是用 Coze Workflow 批量调用 LLM Judge 实现自动化评估。评估体系本身具有可扩展性——评测集扩展到 500 条、1000 条时，评估成本不线性增长。

### 4. 评估→改进的闭环

不是"评完就完了"。每个分类（A/B/C/D）直接映射到对应的改进方向（意图分类器/知识库/答案模板/转人工规则）。Top 3 失败模式分析给出了 P0/P1/P2 优先级和具体的改动方案。评估不是为了证明 Bot 行不行，是为了告诉团队**改哪里**。

### 5. 可审计的决策路径

每条 case 的 classification 都带 decision_path——可以追溯到具体在哪一步触发、因为什么条件停止。v3.1/v3.1s 的 decision_path 进一步标注了 STAR 维度和三问法触发情况。这使得评估结果可以在团队内被 review 和质疑——不是"LLM 说的"，而是"Step 4 Q1+Q2 触发 → D2"。

### 6. 版本选择策略

三个 Judge 版本对应不同使用场景（v3.0 快速原型、v3.1 日常迭代、v3.1s 上线门控），体现了"评估工具的严格度应该匹配使用场景的风险等级"的产品思维。

## 六、AI 相关性总结

本项目体现的 AI PM 能力维度：

| 能力维度 | 项目中的体现 |
|----------|-------------|
| **AI 产品评估体系设计** | 从零推导 MECE 五类决策树，不依赖现成 benchmark |
| **Prompt Engineering** | Judge Prompt 三版本迭代，STAR 清单、三问法、触发即停、锚点内置 |
| **LLM 能力边界理解** | B 类区分"事实错误"和"断言无据"（幻觉），Step 4 判据④区分"AI 能做的"和"AI 不能做的" |
| **用 AI 评估 AI** | Coze Workflow + LLM Judge 实现自动化评估，规模化质量审计 |
| **评估器校准** | 人工标注 vs LLM 输出对比，发现系统性偏差，针对性修复 |
| **数据驱动迭代** | Top 3 失败模式 → 根因归因 → P0/P1/P2 优先级 → 量化预期收益 |
| **AI 产品风险意识** | D 类专门评估"该不该由 AI 回答"，体现对 AI 合规/安全/品牌风险的认知 |
| **跨 LLM 一致性** | v3.1 的 STAR/三问法设计目标之一就是让不同 LLM 跑同一 case 得到一致结果 |
