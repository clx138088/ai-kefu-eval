# 02-MECE 评分标准

> 基于 91 条蓝月亮客服评测集的真人实操推导  
> 配套：`03-Judge-Prompt+Workflow.md`（LLM 执行）、`04-Tracing表.md`（数据记录）

---

## 一、框架总览

一个客服回答可能失败的方式，按检查顺序分为五大类：

```
AI 回答了一个 query
  │
  ├─ ① 搞懂用户在问什么了吗？
  │     ├─ 没有一个显性诉求被正确回应          → A3
  │     ├─ 部分显性诉求没回应                 → A2
  │     └─ 显性诉求全回应了，但停在字面没往前走   → A1
  │     搞懂了（显性全回应 + 捕捉到隐含需求）→ 往下
  │
  ├─ ② 核心结论对吗？（补上缺失信息后结论会被推翻吗？）
  │     ├─ 结论本身是错的，会造成实际损失        → B3
  │     ├─ 结论对但有关键偏差，用户知道真相后会改主意 → B2
  │     └─ 结论对但有细微偏差，用户知道后不会改主意  → B1
  │     结论对且无明显偏差 → 往下
  │
  ├─ ③ 关键信息都给了吗？用户能照着做吗？
  │     ├─ 缺了用户必需的信息，不追问就卡住       → C3
  │     ├─ 缺了用户大概率需要的信息（链接/步骤）   → C2
  │     └─ 缺了锦上添花的信息，不补也行          → C1
  │     信息齐全 + 可操作 → 往下
  │
  ├─ ④ 该 AI 来答吗？（内容没问题的情况下）
  │     ├─ 不该，且涉及安全/赔付/法律 → D2
  │     ├─ 不该，但后果可控 → D1
  │     └─ 该 → E 无问题
```

**原则上一条回答只触发第一个不满足的条件。** 这就是 MECE 的互斥性（ME）。五类覆盖了从理解到输出到决策的全部环节，这就是穷尽性（CE）。

## 二、五大类详细定义

### A 类：意图理解错误

> **AI 没有正确理解用户真正在问什么。**

和 query 的"显性诉求"对比来判：

```
显性诉求 = 用户明明白白说出口的问题 / 要求
隐含诉求 = 用户没直接说，但显性诉求背后指向的真实需求
```

| 等级        | 定义                   | 判据                          | 锚点 case                                                |
| --------- | -------------------- | --------------------------- | ------------------------------------------------------ |
| **A1** 轻微 | 回应了所有显性诉求，但没往前走到隐含诉求 | query 里每个问号都答了，用户看完觉得"对但没用" | A-03："为什么优惠券用不了？"→ AI 列了三条通用规则，但没帮用户排查具体哪条不满足          |
| **A2** 中等 | 遗漏了至少一个用户说出口的诉求      | query 有多个问题/诉求，AI 没有全部回应    | P-10："参加满减吗，什么时候发货？"→ 只答了满减。S-C15："跑偏+急+怎么办？"→ 只查了物流节点 |
| **A3** 严重 | 对所有显性诉求的回应都是错的       | 用户问的每个问题 AI 都没答到点上          | "这个多少钱？"→ 回答成分说明                                       |

**A1 vs A2 的唯一区别**：用户没说的还是用户说了的。不是说"用户没说的就次要"——隐含诉求可以是核心的。这个区分是为了定位根因：A1 指向"缺乏推测能力"，A2 指向"缺乏多意图识别能力"。

---

### B 类：事实信息错误 / 断言无依据

> **意图对了，但回答里的关键信息是错的，或做出了无法验证的断言。**

**B 类涵盖两种情况**：

```
B-事实：可验证的信息是错的
  → "500ml"说成"480ml"、"满300减50"说成"满300减30"
  → 判据：对照知识库/产品信息可验证

B-断言：做出了没有事实依据的断言
  → "绝对安全""不会过敏""100%有效""孕妇完全可用"
  → 判据：AI 给出的结论是否有标注/检测/官方认证作为依据？
     没有 → 即使客观上可能正确，也算 B 类。
     因为 AI 不应该替品牌做安全承诺。
```

**B/C 边界判据**：如果把缺失的信息补上，原来回答的核心结论会被推翻吗？

```
会 → 原来的回答在给错误指引 → B 类
不会 → 方向对、信息对、只是不全 → C 类
```

| 等级        | 定义                     | 判据               | 锚点 case                                                                    |
| --------- | ---------------------- | ---------------- | -------------------------------------------------------------------------- |
| **B1** 轻微 | 事实有偏差或断言轻微越界，但不改变用户决策  | 用户知道真相后，原本的决定不会变 | 事实型："500ml"→"480ml"，差 20ml，用户还是会买                                          |
| **B2** 中等 | 事实偏差会改变决策，或做出了无依据的中等断言 | 用户知道真相后会做出不同选择   | 事实型："满300减50"→"满300减30"，凑单金额不同。断言型："适合所有肤质"——没有检测报告支持                      |
| **B3** 严重 | 核心结论错误或危险断言，会造成实际损失    | 用户跟着做已经产生损失或安全风险 | 事实型："洗衣液漏了"→ AI 说"建议晾干后使用"——破损品不应继续使用。断言型："孕妇绝对安全可用"——替品牌做安全承诺，出了事 AI 负不了责 |

**B 类不只有"信息错了"，还包括"断言没有依据"。** 涉及安全/健康/功效的绝对化表述（"绝对""100%""不会"），如果没有官方认证/检测报告支持，即使客观上可能对，也算 B 类。AI 不应该替品牌做安全承诺。

**B1/B2 判据**：把真实情况告诉用户，用户会改变刚才的决定吗？不会 → B1。会 → B2。

---

### C 类：信息缺失 / 不可操作

> **意图对、事实对，但用户看完没法直接行动或还需要追问。**

| 等级        | 定义           | 判据                  | 锚点 case                                  |
| --------- | ------------ | ------------------- | ---------------------------------------- |
| **C1** 轻微 | 漏了锦上添花的信息    | 用户核心问题已解决，补上更好但不补也行 | P-06："一次倒多少？"→ 说了用量但没附使用视频链接             |
| **C2** 中等 | 漏了用户大概率需要的信息 | 用户可能需要追问才能完成操作      | P-01："多少ml？"→ 说了规格但没附商品链接；用户还得自己去搜       |
| **C3** 严重 | 漏了用户必需的信息    | 用户必须追问，否则无法完成操作     | "怎么退货？"→ 说了"联系客服退货"，但没告诉用户去哪找客服、需要准备什么材料 |

**C 类分级判断——问自己两个问题**：

```
问题一：用户看完这个回答，能不能做他本来想做的事？
  ├─ 不能，操作路径断了，不知道下一步是什么 → C3
  └─ 能 → 问问题二

问题二：用户做完这件事，还需要额外搜/点/问吗？
  ├─ 需要 → C2
  └─ 不需要 → C1
```

| 锚点      | AI 回答      | 能操作？    | 需要额外步骤？ | 等级     |
| ------- | ---------- | ------- | ------- | ------ |
| "怎么退货"  | "联系客服退货"   | ❌ 不知道找谁 | —       | **C3** |
| "多少ml"  | "500ml"    | ✅ 知道规格  | ✅ 要搜商品页 | **C2** |
| "一次倒多少" | "1瓶盖约30ml" | ✅ 知道用量  | ❌ 直接倒   | **C1** |

---

### D 类：处置策略错误

> **内容本身没问题，但回答方式错了——该不该由 AI 来答。**

| 等级        | 定义         | 判据                       | 锚点 case                                                         |
| --------- | ---------- | ------------------------ | --------------------------------------------------------------- |
| **D1** 一般 | 策略错了但后果可控  | 用户多花时间/多等一会儿，无安全/金钱/法律风险 | "多少ml？"→ AI 转了人工——浪费坐席，但无实际伤害                                   |
| **D2** 严重 | 策略错了产生风险敞口 | 用户可能受到身体/金钱/法律伤害         | P-04："宝宝能用吗？"→ AI 答了"可以稀释后"——婴幼儿安全风险。用户要求虚开发票 → AI 配合操作——法律合规风险 |

**触发 D 类的条件**：

以下情况 AI 不应自动回答，应采取转人工或正确拒答：

- 涉及人身安全/健康 → 应转人工
- 涉及赔付/退款/补发 → 应转人工
- 用户情绪信号强烈（承诺/却/投诉/曝光）→ 应转人工
- AI 无法自行完成，需要联系外部方（快递/仓库/支付系统）核实 → 应转人工
- 用户请求触犯法律法规（虚开发票、虚假宣传、侵害他人权益等）→ 应拒答 + 说明法律依据
- 低风险 FAQ 却转了人工 → 该答未答

**注意**：正确的拒答算 E（无问题），不该答却答了或该拒却帮了才算 D。

**D 类不与 A/B/C 重叠**：D 类的前提是"①②③ 检查都通过了"，即内容本身没问题——只是不该由 AI 来答。

**例**：L-C01 "物流卡在6.1一直不动"——AI 理解对了、可以解释可能原因、可以告知流程，但无法自行联系快递公司核实。此时正确处置是："这边马上联系快递为您核实，预计3小时内答复"——而不是 AI 自己编一个解释假装解决了问题。

---

### E 类：无问题

> **①②③④ 全部通过。**

AI 正确理解了用户诉求、给出了准确完整的回答、提供了可执行的行动指引、且处置策略正确。

## 三、完整决策树（评分执行流程）

```
输入：user_query + AI 实际回答

Step 1：意图理解检查
  ├─ query 里每个显性诉求，AI 都有回应吗？
  │     ├─ 一个都没回应 → A3，结束
  │     ├─ 部分没回应 → A2，结束
  │     └─ 全部回应了 → 继续
  │
  └─ 用户没直接说但背后指向的隐含需求，AI 往前多走了一步吗？
        ├─ 没有，停在字面意思 → A1，结束
        └─ 捕捉到了 → 进入 Step 2

Step 2：事实正确性检查（先过 B/C 边界，再定级）
  ├─ 补上缺失/修正错误信息后，回答的核心结论会被推翻吗？
  │     └─ 会 → B 类（进入定级）
  │             ├─ 结论本身是错的，跟着做会有实际损失 → B3，结束
  │             ├─ 有偏差但结论方向对，用户知道真相后会改主意 → B2，结束
  │             └─ 有微小偏差，用户知道真相后不会改主意 → B1，结束
  │
  └─ 结论对且无明显偏差 → 进入 Step 3

Step 3：完整性与可操作性检查
  ├─ 用户看完能直接行动吗？还需要追问或自己去搜吗？
  │     ├─ 必须再问一次才能继续操作 → C3，结束
  │     ├─ 大概率需要再搜/再问（缺链接/入口/步骤）→ C2，结束
  │     ├─ 核心问题已解决，补上更好但不补也行 → C1，结束
  │     └─ 能直接行动 → 进入 Step 4

Step 4：处置策略检查
  └─ 该 AI 来答吗？
        ├─ 不该，涉及安全/赔付/法律/无法自行完成 → D2，结束
        ├─ 不该，但后果仅限浪费时间 → D1，结束
        └─ 该 → E 无问题，结束
```

## 四、MECE 验证

| 检验维度        | 验证  | 说明                                        |
| ----------- | --- | ----------------------------------------- |
| **互斥性（ME）** | ✅   | 决策树按优先级顺序执行，触发即停止，一个 case 只落入一个分类         |
| **穷尽性（CE）** | ✅   | 所有 case 必定按决策树走到一个终点（A1~E），不存在无归属的情况      |
| **A/B 边界**  | ✅   | "补上缺失信息后结论是否被推翻"——明确的二值判据                 |
| **B/C 边界**  | ✅   | 同上                                        |
| **A/D 边界**  | ✅   | A 检查的是"内容是否理解正确"，D 检查的是"理解了且答对了但该不该答"，不重叠 |
| **单/多意图统一** | ✅   | A 类分级基于"显性/隐含诉求覆盖率"，不依赖意图数量               |

## 六、使用说明

### 标注一条 case 的步骤

1. 读 user_query，标出所有显性诉求
2. 读 AI 回答，按决策树 Step 1 → Step 4 逐项检查
3. 触发第一个不满足的点 → 确定分类和等级
4. 记录分类结果和简要理由

### 一次标注 10 条的校准方法

1. 先独立标注 10 条
2. 逐条对比标注结果
3. 对不一致的 case，回到决策树，确认是哪一步的理解不同
4. 若决策树本身不够清晰 → 增加判据或锚点 case
